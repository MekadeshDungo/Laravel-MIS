# Plan: Unify AnimalBiteReport + RabiesReport into BiteRabiesReport

## Context

The system has two separate report models that represent the same concept (an animal bite incident that may involve rabies risk):
- **AnimalBiteReport** ‚Äî older, simpler schema with renamed columns
- **RabiesReport** ‚Äî newer, structured 5-section format

Since every rabies incident starts with a bite, these should be one unified "Bite & Rabies Report". The unified model will follow the RabiesReport schema (structured 5-section format). Old AnimalBiteReport data will be migrated. Reports will auto-assign to assistant_vet with a notification on submission.

---

## Step 1: Fix Notification Model/Migration Mismatch

The `Notification` model uses `user_id` but its migration uses `barangay_user_id`. This must be fixed before we can send notifications.

**Files to modify:**
- `database/migrations/2026_02_12_230008_create_notifications_table.php` ‚Äî rename `barangay_user_id` to `user_id`, update FK to reference `users.id`
- `app/Models/Notification.php` ‚Äî already uses `user_id` in fillable (model is correct, migration was wrong)

---

## Step 2: Create Unified `BiteRabiesReport` Model & Rename Table

**New migration:** `2026_04_02_000001_rename_rabies_reports_to_bite_rabies_reports.php`
- Rename table `rabies_reports` ‚Üí `bite_rabies_reports`

**New model:** `app/Models/BiteRabiesReport.php`
- Table: `bite_rabies_reports`
- Same schema as current `RabiesReport` (5-section format)
- All existing fields from `RabiesReport` fillable
- Add `report_source` field (enum: `client_submission`, `staff_entry`, `migrated`) to distinguish origin
- Relationships: `patientBarangay()`, `barangay()`, `rabiesCase()`, `user()`, `notifications()`
- Keep `generateReportNumber()` static method
- Keep accessors: `vaccination_status`, `current_condition`
- Unified statuses: `Pending Review`, `Under Review`, `Resolved`, `Closed`

**Files to modify:**
- `app/Models/RabiesCase.php` ‚Äî update `rabiesReport()` relationship to point to `BiteRabiesReport`

## Step 3: Data Migration ‚Äî Old AnimalBiteReports ‚Üí Unified Table

**New migration:** `2026_04_02_000002_migrate_animal_bite_reports_to_bite_rabies_reports.php`

Column mapping (old ‚Üí new):
| animal_bite_reports | bite_rabies_reports |
|---|---|
| victim_name | patient_name |
| victim_age | patient_age |
| victim_gender (male/female) | patient_gender (Male/Female ‚Äî capitalize) |
| victim_address + barangay | notes (combined) |
| barangay_id | patient_barangay_id |
| bite_date | incident_date |
| bite_location | bite_site (map values) |
| animal_type | animal_species (capitalize) |
| animal_vaccination_status | animal_vaccination_status |
| animal_owner_name | animal_owner_name |
| status (pending/investigating/resolved) | status (map: pending‚ÜíPending Review, investigating‚ÜíUnder Review, resolved‚ÜíResolved) |
| notes | notes (append) |
| action_taken | notes (append) |
| created_at, updated_at | created_at, updated_at |
| user_id | user_id |

Status mapping:
- `pending` ‚Üí `Pending Review`
- `investigating` ‚Üí `Under Review`
- `resolved` ‚Üí `Resolved`
- `in_progress` ‚Üí `Under Review`
- `closed` ‚Üí `Closed`

Generate a `report_number` for each migrated record (e.g., `BITE-2026-NNNNN`).

- Notification content: "New Bite & Rabies Report: {report_number}" with `related_module` = `bite_rabies_report` and `related_record_id` = report ID

**Modify:** `database/migrations/2026_04_02_000003_add_bite_rabies_report_to_notifications_module_enum.php`
- Add `bite_rabies_report` to the `related_module` enum in the notifications table

**Logic:**
```php
// After report creation:
$assistantVets = User::where('role', 'assistant_vet')->get();
foreach ($assistantVets as $vet) {
    Notification::create([
        'user_id' => $vet->id,
        'title' => 'New Bite & Rabies Report',
        'message' => "Report {$reportNumber} has been submitted and needs review.",
        'related_module' => 'bite_rabies_report',
        'related_record_id' => $report->id,
        'is_read' => false,
    ]);
}
```
